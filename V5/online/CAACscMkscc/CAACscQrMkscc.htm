mkscc Quick Reference

Testing source code in the CAA V5 environment
Quick Reference

Abstract

The mkscc command checks source code for validity against some programming rules.
This command takes as input parameters the type of run and the pattern of sources or frameworks to analyze.
The errors detected are reported in html or text format.

Synopsis

mkscc [-W WSPath] [-a|-lFW FWlist|framework [framework]|module [module]] [-jobs #] [-xmlFilter lg,xmlfile] [-settings lg,XMLFilePath] [-html html_output_dir] [-noFilter] [-h]

Usage

You must specify the source code to analyze. You can specify :

  • -a to check the entire workspace.
  • a framework, a list of frameworks or a list of modules.

You can specify the rules activated during the run :

  • specify to not filter results ( some FW are filtered and don't show all errors)
  • specify your own filter file
  • specify a sub-list of rules using your own setting file

You can specify an output directory for html output. If the directory does not exist, it will be created. If none is specified, the default output will be WS\toolsdata\mkcsc\reporting

Options

The mkscc command accepts the following options:

Option Description
-W path The full pathname of the working workspace if not the current directory.
-h Displays the usage of the command and the help of a list of common options.
-a Considers all frameworks of the current workspace or, if in a framework, all modules of the current framework.
-lFW file The full pathname of a file that lists names, one per line, of frameworks to consider.
-lMod file The full pathname of a file that lists names, one per line, of frameworks to consider.
-jobs number Activates the multi-process mode and sets the number of jobs with parameter value, ignored with '-nomp'.
-nomp Deactivates the multi-process mode, all tasks are performed in the main process.
-noFilter To skip the mkcs filtering.
-settings lg,XMLFilePath Specify your own setting file. Tool language AND XML settings file (lg = cpp or java).
-xmlFilter lg,XMLFilePath Specify your own filter file. Tool language AND XML settings file (lg = cpp or java).
-html dirname A directory name for the reporting output.. If the output directory does not exist, it will be created. dirname will contain a file named "_Index.html", which is the main report file. If this option is not specified, the default output directory is used: WS\toolsdata\mkcsc\reporting

Examples

Let's consider the following Workspace named MyWorkspace.

This Workspace includes two frameworks:
  1. FW1
  2. FW2

Analyzing Every Frameworks

Every source file, of every module, of frameworks FW1 and FW2 will be analyzed, with default check settings. Assuming that tool is launched from E:\MyWorkspace. Output will be generated in E:\Csc.out\_Index.html.

E:
cd E:\MyWorkspace
mkscc -a -html E:\Csc.out

Analyzing Only One Framework

Every source file, of every module, of framework FW1 will be analyzed. Output will be generated in E:\Csc.out\Main.html.

E:
cd E:\MyWorkspace\FW1
mkscc -html E:\Csc.out

If the tool is not launched from the framework directory, you can create a file containing the list of framework to check.

E:
cd E:\MyWorkspace
mkscc -lFW E:\MyWorkspace\myFWList.txt -html E:\Csc.out

With myFWList.txt with the following content:

FW1

Analyzing Only One Module

Only module E:\MyWorkspace\FW1\FW1mod1.m will be analyzed. Output will be generated in E:\Csc.out\Main.html.

E:
cd E:\MyWorkspace\FW1\FW1mod1.m
mkscc -html E:\Csc.out

If the tool is not launched from the module directory, you can like for the FW example create a file with modules to check

E:
cd E:\MyWorkspace
mkscc lMod E:\MyWorkspace\myModList -html E:\Csc.out

Using Your Own Setting File

Let's assume that you wrote your own setting file, E:\Settings_User.xml [1].

If you want to use only this setting file (no delivered setting file will be taken into account):

mkscc -a -settings E:\Settings_User.xml -html E:\Csc.out

Analyzing Only what needs to be analyze

mkscc work by delta, meaning that it will check only what is necessary like mkmk is doing during the build. If you modify a header, mkscc will check the header and all the files having this header in dependency. In order this to work, you need to run a full check of your workspace one time

E:
cd E:\MyWorkspace
mkscc -a
