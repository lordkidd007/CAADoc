mkodt command reference
Quick Reference

Abstract

The mkodt command runs test objects. This article describes the command options, gives usage examples, and the command main error return codes.

  • Synopsis
  • Description
  • Options
  • Examples
  • Diagnostics

Synopsis

mkodt can be run in contextual or descriptive mode:

  • Contextual mode: mkodt is run from a framework directory and applies to test objects of this framework:

    mkodt [-htX] -D [ -s shell | -a ][-html][-xml]

  • Descriptive mode: mkodt is run from any directory and applies to the workspace and frameworks passed using appropriate options:

    mkodt [-hkqStX] [-B debugger -b program] [-d wsabspath [-F framework | -FWList file] | -W wsrelpath | [-a | -L scriptlist1 | -l scriptlist2 | -s ODTName]] [-NOMaxTime] [-NoPert] [-o output] [-p path] [-r rtpath] [-UpdateInit] [-html] [-xml] -ExcludFW fileFWExcluded -PriorityFW fileFWPriority

[Top]

Description

Runs test objects described by XML files that describe either executable modules, bat files, or records delivered in test frameworks. Using its options, mkodt gives the choice of running one test object or all the test objects found either in a given framework or workspace.

mkodt is available with the mkmk commands, and can be accessed after setting the TCK [1].

[Top]

Options

mkodt accepts the following options:

Option Description
-a Replays all the test objects of the current folder. When the current folder is:
  • A workspace, mkodt -a replays all the test objects of the workspace.
  • A framework, or the FunctionTests folder of a framework, mkodt - a replays all the test objects of the framework.
  • Either the TestCases or SwitchTestCases folder, mkodt -a replays all the test objects of the folder.
-B debugger Turns the debugger mode on and specifies the debugger to use. Useful with the -b option. According to the operating system, available debuggers are (first one is the default):
Operating System Debuggers
AIX xldb, dbx, IBM Debugger for AIX (idebug and irmtdbgc)
Windows devenv
-b program Uses the debugger on the given program. This program must be a executable module containing a main, located in the workspace run time view.
-D Sets the display mode for debug purposes, that is, opens a window ("xterm" or "DOS") for interactive testing. This window is opened in the same environment that the one used by the test object, thus it is possible to check for the value of environment variables. Available in contextual mode only.
-d wsabspath Specifies the root folder of the workspace from which to get the test objects. The path to this folder must be absolute. With Windows, this path can be either a UNC path or use a mapped network drive.

Examples:

Path Type Example
Windows UNC Path 
mkodt -s ODTName -d \\lw5psrdsy\home\users\psr\MyWorkspace
Windows Mapped Network Drive Path 
mkodt -s ODTName -d e:\users\psr\MyWorkspace
UNIX Path 
mkodt -s ODTName -d /u/users/psr/MyWorkspace

The folder pathname concatenation [2] where the module or EXE are searched for is set to the workspace designated by the -d option, concatenated to its prerequisite workspaces as declared using the mkGetPreq command. This option sets the ADL_ODT_CONCATENATION environment variable [3] to this value. See also the -p and -r options.
-ExcludFW fileFWExcluded Excludes a list of test framework from a replay. The file fileFWExcluded must have the following format: 
FW1.tst
FW2.tst
FW3.tst

When using this option, pay attention to not use the -L or -l options, and to carefully use the -PriorityFW option.
-F framework Specifies a framework or several frameworks of the workspace. This option is to be used with an option that specifies the test objects to run, that is the -a, the -s, or the -l option.
-FWList file Specifies a file containing a list of frameworks. This option is to be used with an option that specifies the test objects to run, that is the -a, the -s, or the -l option.
-h Displays the mkodt help
-k Keeps the temporary folder set using the ADL_ODT_TMP environment variable [3]. Otherwise, this temporary folder is deleted at the end of the replay.
-L file1 Reads the file file1 containing a list of test objects to execute. These test objects can be in different frameworks of the workspace. file1 must have the following format: 
FW1.tst Type ODTName1
FW2.tst Type ODTName2
FW3.tst Type ODTName3

Where Type can be either TestCases or SwitchTestCases, and ODTNamex is the test object XML file name, without the extension .xml. If the path of file1 is not specified, this file must be either in the current folder or in the workspace root folder.
-l file2 Reads the file file2 containing a list of test objects to execute. file2 must have the following format: 
ODTName1
ODTName2
ODTName3

Where ODTNamex is the test object XML file name, without the extension .xml. If the path of file2 is not specified, this file must be either in the current folder or in the workspace root folder.
-NOMaxTime Disables the supervision of the test object's execution elapsed time. No limit is set. Otherwise, the maximum execution elapsed time is five minutes by default, and can be changed thanks to the ODT_MAX_TIME environment variable [3] up to 60 minutes.
-NoPert Disables the pertinence measure for an ODT. The pertinence step is not executed [4]. You can also use the ODT_PERTINENCE environment variable [3].
-o output Specifies an output folder [5] and sets the ADL_ODT_OUT environment variable [3].

The results of mkodt (ODTName.traces and ODTName.mkodt_results files) are created by default in the <Framework.tst>/FunctionTests/Output/<OS>, where <OS> is the identifier of the operating system used to run mkodt [6].
-p path Specifies the folder pathname concatenation [2] where the module or EXE are searched for, and sets the ADL_ODT_CONCATENATION environment variable to this value [3]. Folder pathnames are separated by a : (colon) with UNIX and a ; (semicolon) with Windows. See also the -d and -r options.
-PriorityFW file3 Reads the file file3 file containing a list of test frameworks the test objects of which are to be replayed as a priority, that is, before the other test objects belonging to other frameworks. file3 file must have the following format: 
FW1.tst
FW2.tst
FW3.tst

If the path of file3 is not specified, this file must be either in the current folder or in the workspace root folder.
-q Quiet mode. Displays in the command prompt only one line per test object replayed, and whether it succeeds or fails.

For each test object replayed, mkodt displays:

## RUNNING ODT [CAASystem.tst CAASysEnvVar] (1 / 1)
elapsed_time: 00 min 06 sec
## ++ CAASysEnvVar : Test execution successful code 0

When using the -q option, mkodt displays:

## CAASystem.tst   CAASysEnvVar : OK

See also the -d and -p options.
-r rtpath Specifies the folder pathname concatenation [2] where the modules or EXE are searched for without setting or changing the ADL_ODT_CONCATENATION environment variable [3] value. Folder pathnames are separated by a : (colon) with UNIX and a ; (semicolon) with Windows.
-S Searches test objects in SwitchTestCases instead of TestCases by default.
-s ODTName Lists the test object(s) to run. ODTName is the test object XML file name, without the extension .xml.

To run a single test object, type:

mkodt -s ODTName

To run several test objects, say three for example, type:

mkodt -s ODTName1 ODTName2 ODTName3
-W wsrelpath Specifies the workspace root directory using a relative path from the $HOME directory. Valid with UNIX only.
-X Forces output display. By default, program's outputs are not displayed, and this option forces them to be displayed
-html Provides with an html view of the results of the replay. Generates an html file tree which gathers the results by framework and for each ODT displays links to its traces and shows you the return code alongside other valuable pieces of information. Html output is located by default in  <Output>/ToolsData/TestAuto/mkodt/<OS>/odt_name_mkodt_result.html, where <OS> is the identifier of the operating system used to run mkodt [6]. You can change this default output folder using the -o option. 
-xml Generates xml mkodt results for every replayed odt. xml data sums up general results and may output additional information that are specific to the replay. Used for collecting results in databases. Location: <Output>/<FW.tst>/FunctionTests/Output/<OS>/odt_name_mkodt_result.xml, where <OS> is the identifier of the operating system used to run mkodt [6]. You can change this default output folder using the -o option.

Examples

  • To run the test objects whose shells are listed in the shell file in contextual mode. 
    mkodt -D -s shell
  • To run, with UNIX only, all the test objects in a given workspace by specifying the workspace root relative directory path. 
    mkodt -a -W MyWorkspace
  • To run, with UNIX only, all the test objects of a given framework of a given workspace by specifying the workspace root relative directory path. 
    mkodt -a -W MyWorkspace -F fwname
  • To run all the test objects in a given workspace by specifying the absolute workspace root directory or folder.
    UNIX 
    mkodt -a -d /u/users/psr/MyWorkspace
    Windows 
    mkodt -a -d e:\users\psr\MyWorkspace
  • To run all the test objects in a given workspace by specifying the absolute workspace root directory or folder, and generate the output in a given folder/directory instead of the default FunctionTests/Output in each framework.
    UNIX 
    mkodt -a -d /u/users/psr/MyWorkspace -o /u/users/psr/output
    Windows 
    mkodt -a -d e:\users\psr\MyWorkspace -o e:\users\psr\output
  • To run all the test objects in a given workspace by specifying the workspace root directory or folder, and generate the output in a given folder/directory instead of the default FunctionTests/Output in each framework, using the executable programs located in a local directory and resetting the ADL_ODT_CONCATENATION variable to this folder/directory.
    UNIX 
    mkodt -d /u/lego/CXR24/BSFTST -a -o /u/users/psr/output -p /u/users/psr/steplib/CXR24/LocalWS
    Windows 
    mkodt -d \\dsone\rnd\CXR24\BSFTST -a -o e:\users\psr\output -p e:\steplib\CXR24\LocalWS

    The common content of the ADL_ODT_CONCATENATION and of the runtime concatenation used is either /u//u/users/psr/steplib/CXR24/LocalWS with UNIX or e:\steplib\CXR24\LocalWS with Windows.

  • To run all the test objects in a given workspace by specifying the workspace workspace root directory or folder, and generate the output in a given folder/directory instead of the default FunctionTests/Output in each framework, using the executable programs located in a local directory without resetting the ADL_ODT_CONCATENATION variable.
    UNIX 
    mkodt -d /u/lego/CXR24/BSFTST -a -o /u/users/psr/output -r /u/users/psr/steplib/CXR24/LocalWS
    WWindows 
    mkodt -d \\dsone\rnd\CXR24\BSFTST -a -o e:\users\psr\output -r e:\steplib\CXR24\LocalWS

    The content of the ADL_ODT_CONCATENATION variable is unchanged and the directory in which the programs are searched for is either /u/users/psr/steplib/CXR24/LocalWS with UNIX or e:\steplib\CXR24\LocalWS with Windows.

Diagnostics

Possible exit status values are:

Code Description
0 Successful completion
>0 Failure because any of the following

The following return code list contains all the known mkodt return codes and all the characteristics RECORD or BATCH return code.

Error Code Returned by Reason
1 RECORD/BATCH
  • CATRecordCapture: error in the path for the CATCommandSelector (the name of the control has changed)
  • Kill in max_time: ODTs reply exceed the defined max_time
  • Server killed.
3 RECORD/BATCH Return error code in batch.
4 RECORD/BATCH ENOVIA: connection with java.
6 RECORD/BATCH CATRecordCapture: data not find at the specified path.
10 RECORD/BATCH Error in the selection agent: object not found.
19 RECORD/BATCH Macro execution problem.
32 RECORD/BATCH Drafting macro problem.
40 mkodt TSTPROVIDER problem during initialization.
45 mkodt Problem during execution of GetVPLMProvider, PLMDBSetupB (for ENOVIA/VPMENV ODT type), ManagedReplay not found (for MANAGED ODT type) or ODT is not compatible with a given ENOVIA server.
46 mkodt No CLIENT/COMMON/SERVER type detected in ODT shell-script.
47 mkodt CTDReplay command: problem during execution
50 mkodt ODTInit phases KO.
51 RECORD/BATCH Bad CATRep management.
55 mkodt Forbidden usage of rm in ODT shell-script (rm authorized on tmp dir only).
56 mkodt Forbidden export of PATH, LIBPATH, SH_LIBPATH, LD_LIBRARY_PATH in ODT shell-script.
58 mkodt Forbidden usage of mkrun ODT shell-script.
66 RECORD/BATCH Geometry-Topology has detected one severe errors.
67 RECORD/BATCH Core Dump: too many calls to Release!
77 RECORD/BATCH Memory leaks.
84 RECORD/BATCH Container not closed, Geometric Modeler memory problem / Object destruction (bad implementation in application).
85 RECORD/BATCH Geometric Modeler memory problem / Object destruction (bad implementation in application).
86 RECORD/BATCH Geometric Modeler memory problem / Object destruction (bad implementation in application).
99 mkodt Kill in max_time (the opening process is too long).
101 RECORD/BATCH Cannot find a dll (Tools/Options on HP).
102 RECORD/BATCH ENOVIA Database connection problem.
126 shell command Shell access permission not correct (chmod 777 *.sh).
127 shell command Library/Module not found.
134 RECORD/BATCH Core dump: visualization (event management, CATPathElement, etc.).
138 RECORD/BATCH Core dump: visualization.
139 RECORD/BATCH Core dump: pointer NULL.
255 RECORD/BATCH Batch ODTs return code.
