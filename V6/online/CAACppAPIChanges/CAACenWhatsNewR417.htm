What's New in 3DEXPERIENCE R2015x
Technical Article

Abstract

New modelers, or enhancements to existing modelers are listed here, compared with the previous release. Some enhancements or maintenance reasons may also require changes regarding tools or methodologies. This article lists them, and points to the corresponding documentation.

Logon to Collaboration Server with CAA Batch Apps

In 3DEXPERIENCE R2015x, to enable Single Sign On, the collaboration server is now deployed with external authentication activated on the 3DPassport authentication server, thus CAA Batch Apps created using the API and that perform their own logon to the collaboration server must use a login ticket instead of the User, Password & Security Context information. This is described in in the Connecting to V6 Server code sample. Please note that most of the other code samples have not been updated yet and so should be modified before use.

UTF8 code page for CATNLS Files

CATNLS message catalogs should now be encoded in UTF-8. See Managing Character Strings for details of the format. It is mandatory to convert CATNLS message catalogs to the new format. This can be very simply accomplished using widely available tools like the scite editor (www.scintilla.org). It is also recommended to move them from CNext folder to a RUNTIME DATA module, as shown in code samples and explained here. Please note however that non-UTF-8 catalogs will be supported for 2 releases.

Changes in Server Objects Basics Attributes Names

The basic attributes are the standard attributes that we can find on any server object and that are used to support the basic operations (transfer ownership, revise, maturity change, lock, etc.). In 3DEXPERIENCE R2015x, we have changed the name of some attributes:

Type holding the attribute Previous name Previous type New name New type
PLMEntity V_user String owner String
V_organization String organization String
V_project String project String
C_created Date originated Date
C_modified Date modified Date
LOCKSTATUS (Y or N) String reserved Boolean
LOCKUSER String reservedby String
PLMCoreReference and PLMCoreRepReference V_maturity String current String
V_version String majorrevision String

Note that for one attribute (LOCKSTATUS), we have also changed its type from string to boolean.

We made this changes to align the server side and client side views of the data model that are used in the customization tools and through the programming APIs.

When you want to access those attributes (by their names) in your code, it is mandatory to use the new names. Please note however that except when we considered it non relevant, API should be upward compatible for the next two releases.

For example CATCkePLMNavPublicServices class is not compatible. It allows to list attributes on a given type, aiming to provide a choice of attribute through the application, directly to the end user. The methods of this class are returning only the new names and this should have no impact on the code of a CAA application.

Changes in Server Objects Type Hierarchy

We changed slightly the type hierarchy in 3DEXPERIENCE R2015x, in order to align the server side and client side views that are exploited by the customization tools and the programming APIs. PLMEntity was the root type for all VPM applications, whereas BusinessType and RelationType were the root types for all Governance applications.

We unified the hierarchies (thus suppressing the distinction between VPM and Governance applications) by:

  1. Introducing the inheritance between PLMEntity and BusinessType: this change being upward compatible.
  2. Making PLMCoreInstance and PLMCoreRepInstance inherit from RelationType instead of PLMEntity. Unfortunately, this change is not upward compatible.

The change of the hierarchy of PLMCoreInstance and PLMCoreRepInstance may impact algorithms based on navigation in type hierarchy.

For example, this sequence of code was valid for reference and instance in R2014x, and will fail for instances in R2015x.

// Retrieving the root type for PLM objects
CATIType_var typePLMRoot;
CATCkePLMNavPublicServices::RetrieveKnowledgeType("PLMEntity", typePLMRoot);
// Verifying that a type is a kind of PLMEntity
if (myType->IsaSortOf(typePLMRoot)) // myType is a CATIType
{
}

You can replace this sequence by this one:

// Retrieving the root type for PLM objects
CATIType_var typePLMRoot;
CATCkePLMNavPublicServices::RetrieveKnowledgeType("DatabaseObjectType", typePLMRoot);
// Verifying that a type is a kind of DatabaseObjectType
if (myType->IsaSortOf(typePLMRoot)) // myType is a CATIType
{
}

Please note that it is very unlikely that this kind of scenario happens in CAA applications.

Deprecation of CATRep Constructors

To prepare future architecture evolutions in the visualization engine, the constructors of 3D inherited classes of CATRep class are being replaced by static methods called CreateRep. This modification does not appear in the Encyclopedia C++ API Deprecated Index and the default constructors are still available in 3DEXPERIENCE R2015x.

How to migrate:

  • Internal allocation: replace calls to default constructor by a call to the factory method. 
    CAT3DBagRep * newRep = new CAT3DBagRep();
    should be replaced by: 
    CAT3DBagRep * newRep = CAT3DBagRep::CreateRep().
  • Allocation on stack: replace by heap allocation using the factory method. 
    void function()
{
 CAT3DBagRep TempRep
 ...
}
    Should be replaced by: 
    void function()
{
 CAT3DBagRep * TempRep = CAT3DBagRep::CreateRep();
 ...
}
  • Allocation for an internal class attribute: replace by a pointer attribute pointing an instance allocated using the factory. 
    class myClass
{
 ...
 private:
  CAT3DBagRep _myRep:
};
    Should be replaced by: 
    class myClass
{
...
 private:
 CAT3DBagRep * _myRep;
};
...
_myRep = CAT3DBagRep::CreateRep();
...

Deprecation of Read/Write Access to Graphical Attributes and Graphic Primitives - Last Release

In relation to visualization performance amelioration operations, it was necessary to enforce the synchronization between user data and visualization caches. For that, methods returning non const pointers to graphical attributes or methods modifying or returning non const pointers to geometric data were deprecated and replaced by methods that enable a better control of visualization modifications. Write access to graphic primitives is allowed through services called "Edit Helper" (for instance CAT3DFaceGPEditHelper). These services check that some conditions are met before allowing write access to graphic primitive data.

Deprecation of Feature Modeler syntax to identify startups extensions

Starting with 3DEXPERIENCER2015x and V5-6R2015, identification of startups extensions using only their late type is deprecated in the API. One of the following syntaxes should now be used for startup identification:

  "`StartUpIdentifier`@`Catalog.CATfct`"
  "`StartUpIdentifier`#<number>@`Catalog.CATfct`"
  "Ou???????-????-????-????????????????@`Catalog.CATfct`"

An example of modification can be found in the Working with Feature Extensions Use Case, the folowing line:

	// 6-1- Instantiate the "CAAOsmNovel" StartUp Facade 
	CATUnicodeString NovelStartUpType("CAAOsmNovel");
	CATFmStartUpFacade NovelStartUpFacade(MyCredential,NovelStartUpType);

has been changed to:

	// 6-1- Instantiate the "CAAOsmNovel" StartUp Facade 
	CATUnicodeString NovelStartUpType("`CAAOsmNovel`@`CAAOsmCatalogSU.CATfct`");
	CATFmStartUpFacade NovelStartUpFacade(MyCredential,NovelStartUpType);
