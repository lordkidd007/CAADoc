Acknowledging Data Exchange through a PLM Channel

Use Case

Abstract

This use case illustrates how to acknowledge data exchange using confirmation file through PLM Channel.

A PLM Channel is an exchange context created between two specific repositories in order to allow data exchange and ownership transfers. Refer the Technical article [1] for the details of PLM Channel creation and Data Exchange.

What You Will Learn With This Use Case

This use case is intended to show you how to acknowledge the data exchange using the confirmation file ( generated during synchronizing the PLM Channel [6]).

Acknowledge updates the synchronization status at Sender site, based on the confirmation file received from Receiver, completing the data exchange transaction. It generates a Report file with the synchronization status from the receiver site.

The CAASrmAcknowledgeExchangeThroughChannel Use Case

CAASrmAcknowledgeExchangeThroughChannel is a use case of the CAASRMExposedServices.edu framework that illustrates PLMSRMExposedServices framework capabilities.

What Does CAASrmAcknowledgeExchangeThroughChannel Do

Synchronizing data exchange [6] use case has generated a confirmation file stored into a PLM Briefcase file . This file has been sent (through email, file transfer etc) to the current site . Using this confirmation file , the current site process the final step of Data Exchange between the two sites by generating a report (with synchronization status from the other site).

How to Launch CAASrmAcknowledgeExchangeThroughChannel

To launch CAASrmAcknowledgeExchangeThroughChannel, you will need to set up the build time environment, then compile CAASrmAcknowledgeExchangeThroughChannel along with its prerequisites, set up the run time environment, and then execute the use case [7].

To launch the use case execute the command:

mkrun -c "CAASrmAcknowledgeExchangeThroughChannel Repository Server User Password SecurityCtx AcknowledgementFilePath"

where:

Repository
Server The server as Host::port_number/RootURI
User The user name
Password The user password to authenticate the user
SecurityCtx A string representing the security context (Role.Organization.Project). Choose Role as Designer
AcknowledgementFilePath The path of the confirmation file generated during the synchronization of the PLM Channel at remote site .It is required to update the synchronization status at the current site. The file must exist and its extension must be ".PLMBriefcase". Example: "C:\Temp\ConfirmationFilePath.PLMBriefcase".

Where to Find the CAASrmAcknowledgeExchangeThroughChannel Code

The CAASrmAcknowledgeExchangeThroughChannel use case is made of a single file located in the CAASrmAcknowledgeExchangeThroughChannel module of the CAASRMExposedServices.edu framework:

InstallRootFolder\CAADoc\CAASRMExposedServices.edu\CAASrmAcknowledgeExchangeThroughChannel.m\

where InstallRootFolder [8] is the folder where the API CD-ROM is installed.

Prolog

The CAASrmAcknowledgeExchangeThroughChannel use case begins by creating a session and connecting to the Input Repository with an appropriate role. The use case "Creating a Basic PLM Batch" [8] deals with Opening and Closing a PLM Session.

Acknowledge the data exchange

When the sender site receives the confirmation file (generated during synchronizing the data exchange at the receiver site [6]), sender will then acknowledge the data exchange (the final step of Data Exchange process).

...
   CATPLMExchangeChannelServices* pExchangeChannelServices = 
   			      new CATPLMExchangeChannelServices();
   if (NULL!=pExchangeChannelServices )
   {
      ...
      HRESULT Rc = pExchangeChannelServices ->AcknowledgeExchange
      			              (iacknowledgementFile,ochannelId,oreport);
      ...
   }
...

AcknowledgeExchange takes three arguments:

  • iacknowledgementFile, CATUnicodeString type, path of the confirmation file received from the receiver site. The file must exist and its extension must be ".PLMBriefcase".
  • ochannelId, CATString type, the channel identification between Sender and Receiver. The same ochannelId was used by the Sender site while publishing the PLM data [5]
  • oreport, CATUnicodeString type, the report generated with the synchronization status from receiver site.

A call to CATPLMExchangeChannelServices::AcknowledgeExchange API generates a report file with the synchronization status from the receiver site. This call completes the data exchange process between the two sites.

Epilog

The CAASrmAcknowledgeExchangeThroughChannel use case ends by simply closing the PLM session. Since all use cases necessarily conclude by closing the PLM session, we have a dedicated article on this topic namely "Creating a Basic PLM Batch" [8].
