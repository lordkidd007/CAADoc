Browsing a Validation
Use Case

Abstract

This article discusses the CAAValBrowsingValidation use case. This use case explains how to navigate an existing Validation using APIs of the Validation Modeler belonging to the ENOValidationInterfaces framework.

What You Will Learn With This Use Case

The primary objective of this use case is to familiarize the user with the APIs meant for browsing a Validation. Listed below are the topics that a user gets an introduction to, through this use case.

  • Opening and Loading a Validation in various modes based on what components need to be retrieved/browsed
  • Various APIs of the ENOValidationInterfaces framework to browse the various entities of a Validation

The CAAValBrowsingValidation Use Case

CAAValBrowsingValidation is a use case of the CAAValidation.edu framework that illustrates the ENOValidationInterfaces frameworks capabilities.

What Does CAAValBrowsingValidation Do

The input PLM Data for the Use Case is a Validation.

The use case loads the Validation Reference and its Child in session.

The Use Case execution results in an output text which gather all information about the Validation. A sample of output can be seen below.

How to Launch CAAPstBrowseProductModel

To launch CAAValBrowsingValidation, you will need to set up the build time environment, then compile CAAValBrowsingValidation along with its prerequisites, set up the run time environment, and then execute the use case [1].

To launch the use case execute the command:

mkrun -c " CAAValBrowsingValidation Repository Server User Password SecurityCtx Environment PLM_ExternalID V_version"

where:

Repository
Server The name and port of the server as name:port_number
User The user name
Password The user password to authenticate the user
SecurityCtx a string representing the security context ( Role. Organization. Project). Choose Role as Reviewer ( as the use case does not modify the PLM Data)
Environment A PLM environment containing PLMProductDS (a PRODUCT's custo)
PLM_ExternalID The PLM_ExternalID of a PLM Reference or PLM Representation Reference value of the input model
V_version The V_version of a PLM Reference or PLM Representation Reference

where InstallRootDirectory [1] is the directory where the CAA CD-ROM is installed.

Validation does not exist in 3D XML format: there is no delivered sample data to test.

Where to Find the CAAValBrowsingValidation Code

The CAAValBrowsingValidation use case is located in the CAAValBrowsingValidation.m module of the CAAValidation.edu framework:

InstallRootDirectory\CAAValidation.edu\CAAValBrowsingValidation.m\

where InstallRootDirectory [1] is the directory where the CAA CD-ROM is installed.

Step-by-Step

Following are the main steps in CAAValBrowsingValidation:

  1. Prolog
  2. Open the Validation
  3. Browse the Validation
  4. Epilog

Prolog

The CAAValBrowsingValidation use case begins by creating a session and connecting to the Input Repository. This, being the most fundamental prerequisite for all batch use cases, it is the first step in their execution.

In this case, we create a session & connect to Repository with the Reviewer Role (since we ain't modifying PLM data). There are other parameters as well, details of which can be found in the "Creating a Basic PLM Batch" use case article [2]. This article primarily deals with Opening and Closing a PLM session. 

int main(int iArgc, char *iArgv[])
{
	HRESULT HR = S_OK;

	//===========================================================================
	// 1. Check for minimum number of arguments

	const int NbMinArgs = 9;

Browse the Validation

The CAAValBrowsingValidation use case executes the The BrowseValidation Global Routine to compute the output text from the opened Validation and remove it from bag to manage its life cycle using RemoveRoot. 

...

	//===========================================================================
	// 4. Browse the Validation
	//===========================================================================
	HR = BrowseValidation(piValidation);
	if (FAILED(HR))
	{
		cerr << "ERROR during browse" << endl;
		return 2;
	}

	rootsBag.RomoveRoot(piValidation);
	if (NULL != piValidation)
	{
		piValidation->Release();
		piValidation = NULL;
	}

Epilog

The CAAValBrowsingValidation use case ends by simply closing the PLM session. Since all batch use cases necessarily conclude by closing the PLM session, we have a dedicated article on this topic namely "Creating a Basic PLM Batch" [3].

	//===========================================================================

	//===========================================================================
	HR = ::CAAAdpCloseSession();
	if (FAILED(HR))
	{
		cerr << "ERROR during session closing" << endl;
		return 2;
	}

	return 0;
}

The OpenValidation routine takes 6 arguments:

  • CATString type, the repository name
  • CATString type, the environment name
  • CATUnicodeString type, the Validation's name
  • CATUnicodeString type, the Validation's version
  • CATIVALValidation type, the pointer on opened Validation

The OpenValidation global routine has fundamentally four steps:

  • Retrieve the environment
  • Retrieve the type
  • Query the database
  • Expand and open
