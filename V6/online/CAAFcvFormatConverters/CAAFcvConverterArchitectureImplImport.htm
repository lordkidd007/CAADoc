Mechanical CAD Converter Architecture for Import

The following pictures illustrate converters integration and architecture in the ENOVIA V6 Collaborative Store for the Import cases
The Format converters architecture does not depend on whether the Native app files are stored in a PDM or Not.The table on the left is highlighting the components to be developed with the CAA APIS, and the components available in the CAA InfrastructureThe Binding to xPDM Exchange and File Exchange are Services Provided with the CAA infrastructure.

The Main job to do in CAA development is to implement an xApp format Reader.

The detailed architecture of an xAPP convertor for importing to V6 is as follows

It relies on two major components that communicate thru CAA interfaces.

The reader: it is to be implemented by CAA
The reader is composed of a set of components which implement the CAA interfaces for each domain of the Mechanical CAD: Product structure, Boundary Rep, Tessellated Rep, etc,

Remark:
Depending on the capabilities of the format, and the expected capabilities of the translator, only a part of them may be implemented.
For instance: some formats do not embed any tessellated representation; on the contrary, some formats only embed a tessellation representation of the geometry. Some other format do not represent Product Structures or Assemblies.


The writer: it is provided by Format Conversion Services
The writer is composed of a set of translator components which use the CAA interfaces to query the Document , browse its contents, and extract the relevant information to be converted to V6.


API principles:
The APIS are CAA Interfaces, which Name is CATIExchange+”Function”, Function being related to the Domain / Intellectual Property covered by the Api.
These APIS are “get” APIS, or query APIS.
They retrieve the Contents of the Document regarding the Functional Domain or Intellectual Property they cover.
