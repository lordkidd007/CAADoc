Mechanical CAD Converter Architecture for Export


The following pictures illustrate converters integration and architecture in the ENOVIA V6 Collaborative Store for the Export cases.

The Format converters architecture does not depend on whether the Native app files are stored in a PDM or Not.

The table on the left is highlighting the components to be developed with the CAA APIS, and the components available in the CAA Infrastructure.

The Binding to xPDM Exchange and File Exchange are Services Provided with the CAA infrastructure.

The Main job to do in CAA development is to implement an xApp format writer.

 

 

The detailed architecture for an xAPP convertor for Exporting from V6 is as follows

It relies on two major components that communicate thru CAA interfaces.

The CAA interfaces are “get” APIS.

The reader: It is Provided by Format Conversion Services
The reader is composed of a set of components which implement the CAA interfaces for each domain of the Mechanical CAD: Product structure, Boundary Rep, Tessellated Rep, etc,

The Writer: it is to be implemented by CAA
The writer is composed of a set of translator components which use the CAA interfaces to query the Document , browse its contents, and extract the relevant information to be converted to V6. It is to be implemented, as part of the CAA development. The Writer translates the contents to the output Format.

API principles:
The APIS are CAA Interfaces, which Name is CATIExchange+”Function”, Function being related to the Domain / Intellectual Property covered by the Api.
