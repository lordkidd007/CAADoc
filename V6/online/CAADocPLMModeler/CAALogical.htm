Description

This section first presents the objects representing the Logical modeler. Then it describes how this object model view has been translated towards PLM objects based on PLM Core Modeler objects. You will see the description of each PLM package defining these PLM objects.

Conceptual Model

The Requirement (R), Functional (F), Logical (L) and Physical (P) domains allow the whole definition of a product. In other words, it is used for the Product Conception Approach (Product Concept Creation) based on the description of the product through different levels of abstraction (RFLP).

Within RFLP domains, the Logical domain enables to define HOW to engineer the product to satisfy the functional systems (i.e. services) specified in Functional domain. Logical systems and links between logical systems through their ports specify the network.

This is summarized in the following UML schema:

Fig.1 Logical Model

  • Logical domain gathers:

    • Logical System: Hierarchical assemblage of interconnected sub-systems. Systems can be re-used (instantiated).
    • Logical Connection : links ports of systems, i.e. the explicit exchange of data between systems.
    • Logical Port: Specifies the links between ports of systems. There are two kinds of logical ports: causal (In, Out and In/Out direction) and a-causal (no direction).
    • System Type: Information associated to a logical port to specify the type of data going through that port.
    • Logical Communication (also called MUX (Multiplexer) and DEMUX (De-multiplexer)). When defining structured Types, it is necessary to provide constructor and destructor of such type. The logical MUX (Multiplexer) and DEMUX (De-multiplexer) are used for this purpose.
    • Logical Representation: enables to store private data such as picture, simulation data result, etc.

    In addition 3D definition can be added to Logical systems:

    • Logical 3D Representation: to associate 3D representation to a logical system to offer easy-to-use 3D modeling for system reservations, enabling system architects to validate the system architecture early on in the process.
    • Logical 3D Publication: Use for design purpose to expose 3D private data of the representation (e.g. Plan).

Customization

Customization first deals with the modeler PLM package customization [2]. It consists in to create a new package containing new PLM classes for each "Customizable" PLM class of the modeler PLM package. Then, you should take into account the UI masks creation to create either a new security mask file, or to update the default one [3]. Finally, you can integrate your own business logic on each new PLM classes by implementing provided PLM opening ID [4].

UI Mask Customization

When a PLM attribute is defined (inside modeler PLM package) some features like its editability criteria, mandatory /optional option, can be overwritten by UI mask. You can say this PLM attribute is not writable in Query context, this PLM attribute is mandatory in Create context, this user PLM attribute is not never visible and so one. A set of UI masks form a security mask file. We say security mask because it is associated with these files security information. For a people, and a given context can be associated a security mask.

For each customized Modeler PLM package, at least one UI mask file must be created (for the default security mask ), since it must contain the new PLM Attributes. When you use the tool to create a customization, a default UI mask file is provided taken into account the added PLM attributes. You can modify the default file to introduce your change.

Rules to respect for a new UI mask creation

No specific rule to customize UI mask file.
