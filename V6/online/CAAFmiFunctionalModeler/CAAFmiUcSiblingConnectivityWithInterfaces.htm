Abstract

This use case creates Sibling Connection i.e. it creates the connection between two sibling (same level) interfaces (Flow Exposition Instance).

What You Will Learn With This Use Case

The learning's from this Use Case are:

  • How to create the Sibling connection (same level) between two interfaces.
  • Objects involved while creation of connection.

The CAAFmiSiblingConnectivityWithInterfaces Use Case

CAAFmiSiblingConnectivityWithInterfaces is a use case of the CAAFunctModelerUseItf.edu framework that illustrates CATFunctionalModelerUseItf framework capabilities.

What Does CAAFmiSiblingConnectivityWithInterfaces Do

In this use case we have created Sibling connection. Sibling connection means the connection between interfaces those are on the same level.

Initially in this use case we retrieve four objects to form connection:

  • Instance 1 under which the Source Interface is aggregated (Functional Instance 1).
  • Source Connection Point 1 (Flow Exposition Instance Emission).
  • Instance 2 Under which the Target Interface is aggregated (Functional Instance 2).
  • Target Connection Point 2 (Flow Exposition Instance Consumption).

Both Flow Exposition Instances are instantiating the same Functional Flow Reference in order to be compatible for a connection.

How to Launch CAAFmiSiblingConnectivityWithInterfaces

To launch CAAFmiSiblingConnectivityWithInterfaces, you will need to set up the build time environment, then compile CAAFmiSiblingConnectivityWithInterfaces along with its prerequisites, set up the run time environment, and then execute the use case [2].

To launch the use case execute the command:

mkrun -c "CAAFmiSiblingConnectivityWithInterfaces Repository Server User Password SecurityCtx PLMType IDSetAttribute1 IDSetAttributeValue1 [IDSetAttribute1* IDSetAttributeValue1* ...]"

where:

Repository
Server The server as ServerName:ServerInterface/rootURI
User The user name
Password The user password to authenticate the user
SecurityCtx A string representing the security context (Role.Organization.Project). Choose Role as Designer.
PLMType The String representing PLMType
IDSetAttribute1* First Identification Set(IDSet) attribute for the Functional Reference
IDSetAttributeValue1* First IDSet attribute value for the Functional Reference

* We could have several such Attribute-Value set. The count is determined by the IDSet attributes defined by the PLM type of the Functional Reference to be opened.

CAAFmiSiblingConnectivityWithInterfaces.3dxml is the 3D XML file contains the root Functional Reference with the following attributes:

  • PLM_ExternalID: fct00000001_E6ECA968-0000-43B4-53972EE600077943
  • Version: ---
  • V_Name: CAAFmiSiblingConnectivityWithInterfaces_Root

Attention: This 3dxml have been created with OneClick and Unified Typing activated (default environment).

Attention: Please refer to the document [1] for how to import and reimport such a 3D XML file in your PLM repository.

Step-by-Step

There are several logical steps in CAAFmiSiblingConnectivityWithInterfaces use case.

  1. Prolog
  2. Retrieving the two Functional Instances as Functional Connectable
  3. Retrieving associated Functional Connection Point: Flow Exposition Instances between which the sibling connection is to be made
  4. Creating Connection between two Instances through Interfaces
  5. Epilog

Prolog

We connect to the input repository and a PLM session is created. A PLM session is an environment essential for running a CAA use case. The article [1]delves further into the process of building and launching a CAA use case.

Here we open in Authoring mode [3] the Functional structure input provided by the use case . This input model contains two instances with Interfaces.

Retrieving the two Functional Instances as Functional Connectable

In this step we retrieve two objects from the input Functional structure. The objects retrieved are the two instances in between we want to create connection:

  • Instance 1 under which the Source Interface is aggregated (piFctConnectableOnFuncInst_1 as a CATIFctConnectable interface pointer).
  • Instance 2 Under which the target Interface is aggregated (piFctConnectableOnFuncInst_2 as a CATIFctConnectable interface pointer).

Retrieving associated Functional Connection Point: Flow Exposition Instances between which the sibling connection is to be made

In this step we retrieve two objects from the input Functional structure. The objects retrieved are the corresponding Interfaces through which connection is made:

  • Source Interface 1 (pFctConnectionPointOnConnectionPoint_1 as a CATFctConnectionPoint pointer).
  • Target Interface 2 (pFctConnectionPointOnConnectionPoint_2 as a CATFctConnectionPoint pointer).

Creating Connection between two Instances through Interfaces

Now use case creates a connection between two instances through retrieved Interfaces.

...
    CATIType_var spType = NULL_var;
    CATIFctConnection *opiFunctionalConnection = NULL;
    hr = piFctConnectableOnFuncInst_1->Connect(spType,
		pFctConnectionPointOnConnectionPoint_1,
		piFctConnectableOnFuncInst_2, 
		pFctConnectionPointOnConnectionPoint_2,
		opiFunctionalConnection);
...

A call to CATIFctConnectable::Connect creates connection between two connectables, pFctConnectionPointOnConnectionPoint_1 is an Interface aggregated under the instance piFctConnectableOnFuncInst_1 on which we have made a call for connect API. piFctConnectableOnFuncInst_2 is an other instance to which we want to connect first one. And pFctConnectionPointOnConnectionPoint_2 is the other Interface aggregated under second instance.

A Connectable can be a Functional Instance or a Functional Reference. Here CATIFctConnectable::Connect is used for Sibling Connectivity, but it can also being used to create Parent-Child Connectivity (one of the Connectable is a Reference) [4].

CATLogConnectionPoint can be used for Interfaces and Ports. So, CATILogConnectable::Connect can also being used to connect Ports [5].

Here we pass CATIType as NULL_var this will select the appropriate Functional connection type. The API will return Connection object.

Epilog

The CAAFmiSiblingConnectivityWithInterfaces use case ends by simply closing the PLM session. Since all use cases necessarily conclude by closing the PLM session, we have a dedicated article on this topic "Connecting to V6 Server" [2].

In Short

This use case basically illustrated the user how to create the Sibling Connection (using CATIFctConnectable::Connect API). Sibling connection means the connection between Interfaces those are on the same level. The four objects required to form sibling connection are:

  • Instance 1 under which the Source Interface is aggregated.
  • Source Interface (Flow Exposition Instance 1).
  • Instance 2 Under which the Target Interface is aggregated.
  • Target Interface (Flow Exposition Instance 2).
