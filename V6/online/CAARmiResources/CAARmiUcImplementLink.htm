Assigning an Operation to a Resource Occurrence
Use Case

Abstract

This article describes an example of use of Framework DELPPRResourceItf

This usecase shows how to assign a operation occurrence to a Resource occurrence; how to navigate along occurrences linked by this relationship; and how to unassign the operation from the resources.

What You Will Learn With This Use Case

In this use case, you will learn how to use the DELIPPRResourceAuth and DELIPPRResourceNav interfaces in order to:

  • Assign a operation occurrence to a Resource occurrence.
  • Navigate on assignation relations from Resource to Operation or from Operation to Resource.
  • Unassign the operation from the resource occurrences.

The CAARmiImplementLink Use Case

CAARmiImplementLink is a use case of the CAAPPRResourceItf.edu framework that illustrates DELPPRResourceItf framework capabilities.

What Does CAARmiImplementLink Do

  • Retrieving the DELIPPRResourceAuth and DELIPPRResourceNav interfaces.
  • Retrieving the DELIPPRResourceAuth and DELIPPRResourceNav interfaces.
  • Querying Root Resource and root System references in the database.
  • Assigning the Operation child occurrence to the Resource child occurrences.
  • Assigning the Operation occurrence to Resource occurrences.
  • Browsing assignments, getting a specific resource from an operation and operations from resource.
  • Operation is assigned to a Resource instance in respect with the first upward scope.
  • Unassign the assigned Operation from all the Resource occurrences.

The ability to assign is checked before assignment Creation. For a description of a comparable scope dependency see [6] CAAFmiImplementLink use case.

Use case Objects: CAARmiImplementLink

Expected Result:
The result is a return code, its value is:

  • 0 on successful completion/
  • 1 if execution failed.

How to Launch CAARmiImplementLink

To launch CAARmiImplementLink, you will need to set up the build time environment, then compile CAARmiImplementLink along with its prerequisites, set up the run time environment, and then execute the use case [1].

Additional instructions:

This use case is using datas created by other use cases: [3] and [4], that you should run first.
mkrun -c "CAARmiImplementLink Repository Server User Password SecurityCtx Environment"

where:

Parameter description
Repository The name of the repository (or provider)
Server The name and port of the server
User The user identifier
Password The user password
SecurityCtx A string representing the security context (Role.Organization.Project)
Environment A PLM Environment containing a modeler customization

Example:

mkrun -c "CAARmiImplementLink PLM1 ve4al10dsy:8090 uxodtipt Ovb2pcds VPMADMIN.ADMIN.DEFAULT VPM"

Step-by-Step

There are six logical steps in CAARmiImplementLink:

  1. Retrieving the DELIPPRResourceAuth and DELIPPRResourceNav Interfaces
  2. Querying Root Resource and Root System References in the Database
  3. Retrieving Occurrences of Root Resource and Root Operation
  4. Assigning the Operation Child Ooccurrence to the Resource Child Occurrences
  5. Browsing Assignments, Getting a Specific Resource from an Operation and Operations from Resource
  6. Unassigning the Implement Links Created between the Resource Occurrences and the Operation

Retrieving the DELIPPRResourceAuth and DELIPPRResourceNav Interfaces

DELIPPRResourceAuth_var hPPRResourceAuth;
rc = ::GetPPRResourceAuth(hPPRResourceAuth);
            
DELIPPRResourceNav_var hPPRResourceNav;
rc = ::GetPPRResourceNav(hPPRResourceNav);

The DELIPPRResourceAuth interface will later be used to create relation between Operation and Resource and the DELIPPRResourceNav interface to navigate along it.

Querying Root Resource and Root System References in the Database

CATOmbLifeCycleRootsBag Bag;
CATAdpAttributeSet AttributeSetResource;
AttributeSetResource.AddAttribute("PLM_ExternalID","Organizational.1"); 
AttributeSetResource.AddAttribute("V_version","---"); 

CATIType_var spTypeResource;
rc = CATCkePLMNavPublicServices::RetrieveKnowledgeType("VPMReference",spTypeResource);

CATIPLMNavReference * piRootResource = NULL;
int ret = CAARmiFindServices::FindReference( piEnv, Bag, AttributeSetResource, spTypeResource, IID_CATIPLMNavReference, (void **)&piRootResource);

CATAdpAttributeSet AttributeSetSystem;
AttributeSetSystem.AddAttribute("PLM_ExternalID","GeneralSystem.1"); 
AttributeSetSystem.AddAttribute("V_version","---"); 

CATIType_var spTypeSystem;
rc = CATCkePLMNavPublicServices::RetrieveKnowledgeType("DELLmiGeneralSystemReference",spTypeSystem);

CATIPLMNavReference * piRootSystem = NULL;
ret = CAARmiFindServices::FindReference( piEnv, Bag, AttributeSetSystem, spTypeSystem, IID_CATIPLMNavReference,(void **)&piRootSystem);

The DELIPPRResourceAuth interface will be later used to create relation between Operation and Resource and to navigate along it.

Retrieving Occurrences of Root Resource and Root Operation

CATIPLMNavOccurrence_var hNavOccOnRootResource;
rc = CAARmiOccurrenceServices::GetOccurrenceFromResourceReference(hNavRefOnRootResource, hNavOccOnRootResource);
            
CATIPLMNavOccurrence_var hNavOccOnOperationInst;
rc = CAARmiOccurrenceServices::GetOccurrenceFromSingleSystemChild(hNavRefOnRootSystem, hNavOccOnOperationInst);

The custom services "CAARmiOccurrenceServices::GetOccurrenceFromSingle[Resource|System]Child(..)" and CAARmiOccurrenceServices::GetOccurrenceFrom[Resource|System]Reference are used to retrieve the occurrence of the single child of the Resource or Operation.
See next step for some explanations.

Assigning the Operation Child Ooccurrence to the Resource Child Occurrences

For each Root Resource Children, Retrieving its Occurrence and assigning the root Operation's child-Occurrence to it. DELIPPRResourceAuth is used to create Assignments. Specific "where" "who" and "with" assignments are created by DELIPPRResourceAuth::AssignOperation given the Resource category type.

  1. Building the Resource Child-Instances List
  2. Building the Resources Occurrences List
  3. Assigning the Operation Occurrence to each Resource Occurrences

Building the Resource Child-Instances

CATListPtrCATIPLMNavEntity listChildren;
rc = hNavRefOnRootResource->ListChildren(listChildren, 0, NULL);

Building the Resources Occurrences List

CATIPLMNavOccurrence_var hNavOccOnResourceInst[4];	//size of array is the expected number of elements + 1 as we don't use index 0
for (int i=1; i<=3; ++i){
	CATIPLMNavInstance_var hInst = listChildren[i];
	rc = CAARmiOccurrenceServices::GetOccurrenceFromResourceInstance(hInst, hNavOccOnRootResource, hNavOccOnResourceInst[i]);
}

The Resources Occurrences List is found by getting Resource Occurrence children from the root Resource Occurrence.

Assigning the Operation Occurrence to each Resource Occurrences

rc = hPPRResourceAuth->AssignOperation(hNavOccOnResourceInst[i], hNavOccOnOperationInst);

For each Root Resource Children Occurrence, assigning the root Operation's child-Occurrence.
DELIPPRResourceAuth performs this assignment. The implement link between the given Operation and this Resource is created if it does not exist yet. Otherwise, it is modified. The creation or the modification is performed according to "this" Resource category type (MPResourceType).

If the Resource category type is:

  • Organizational, then the "Where" link is set.
  • Working, then the "Who" link is set.
  • NotWorking, then the "With" link is set.

Browsing Assignments, Getting a Specific Resource from an Operation and Operations from Resource

DELIPPRResourceNav is used to Navigate Assignments relations. Specific "where" "who" and "with" assignments may be followed from operation to resource and from resource to operations by specific DELIPPRResourceNav methods.

  1. Getting Operation and Resource "Where" Assignment
  2. Getting Operation and Resource "Who" Assignment
  3. Getting Operation and Resource "With" Assignment
  4. Navigating in the Other Direction: Getting Resource and Operations Assignments

Getting Operation and Resource "Where" Assignment

CATIPLMNavOccurrence * piNavOccOnWhereResource = NULL;
rc = hPPRResourceNav->GetWhereResource(hNavOccOnOperationInst, (void **)&piNavOccOnWhereResource);

Getting Operation and Resource "Who" Assignment

CATIPLMNavOccurrence * piNavOccOnWhoResource = NULL;
rc = hPPRResourceNav->GetWhoResource(hNavOccOnOperationInst, (void **)&piNavOccOnWhoResource);

Getting Operation and Resource "With" Assignment

CATListPtrCATIPLMNavOccurrence listWithResourceOcc;
rc = hPPRResourceNav->GetWithResources(hNavOccOnOperationInst, listWithResourceOcc);

Navigating in the Other Direction: Getting Resource and Operations Assignments

for (int i=1; i<=3; ++i)
{
	CATListPtrCATIPLMNavOccurrence listOperationOcc;
	rc = hPPRResourceNav->GetAssignedOperations(hNavOccOnResourceInst[i], listOperationOcc);
}

Retrieves the list of Operations Assigned to each Resource.

Unassigning the Implement Links Created between the Resource Occurrences and the Operation

rc = hPPRResourceAuth->UnAssignOperation(hNavOccOnResourceInst[i], hNavOccOnOperationInst);

DELIPPRResourceAuth is used to perform the implement link removal on each of the resources.

In Short

Operations may be assigned to a Resource occurrence via Implements Links and navigating between them is made possible through the DELIPPRResourceNav Class. DELIPPRResourceAuth is used to create Assignments. DELIPPRResourceNav is used to Navigate Assignments relations. Specific "where" "who" and "with" assignments are created by DELIPPRResourceAuth::AssignOperation given the Resource category type, and may be followed from operation to resource and from resource to operations by specific DELIPPRResourceNav methods.
