Introduction

This article explains to you how you can navigate through the Instance/Reference Model, the Occurrence Model, and the two models. As usual the purpose are illustrated with the our fetish product, the skateboard.

Fig.1 Skateboard

This model contains:

  • 1 skate containing 1 top (purple) + 2 axles (gray).
  • 1 axle containing 2 wheels (green).
  • 1 wheel containing 3 screws (blue).

Instance/Reference Model Navigation

The instance/reference model is the persistent database model, the one that you retrieve when you load in session a root product. It is based on the PLM Core model, including the six PLM Core objects: PLM Reference, PLM Instance, PLM Instance Representation, PLM Representation Reference, PLM Port and PLM Connection. The following picture shows the relationship between these six PLM objects.

Fig.2 Instance/Reference Model

The current chapter gives the APIs to navigate on this structure, except the connections since the only one APIs to retrieve them are applicative APIs.

The picture below shows the instance/reference model for the skateboard.

Fig.3 Skateboard Model

On this picture you retrieve the four (*) types of PLM Core objects:

  • PLM References are represented in purple/blue boxes- there are four PLM References
    • The skateboard (CAAPstSkateBoard - the root product) , the axle (CAAPstAxleAss) , the wheel (CAAPstWheel) and the screw (CAAPstScrew).
  • PLM Instances are represented in blue oval boxes - Each PLM Instance is an instance of a PLM Reference - From left to right, you retrieve

(*) The input model does not contain port and connection.

Directions of Navigation

The following picture shows the four types of direction (red arrows) to navigate on the instance/reference model.

Fig.4 Direction of Navigation

The meaning of the letter/number within the blue balloons is as follows:

  • P: The navigation starts from a Product (Reference or Instance).
  • R: The navigation starts from a Representation (Reference or Instance).
  • 1: A navigation from a child (an instance) to its parent (a reference).
  • 2: A navigation from a parent (a reference) to its child(s) (instances).
  • 3: A navigation from an instance to its reference.
  • 4: A navigation from a reference to its instance(s).

Example: (1R) means a navigation from a Representation child to its parent, i.e. from a PLM Rep Instance to its aggregating PLM Reference.

These such symbols will be re-used in the next sections to make the relation between the need and [Fig.4]. Before the question: how to get the PLM Rep Ref aggregating a PLM Rep Instance, (1R) will be specified.

Navigation APIs

The interfaces to use are those of the CATPLMComponentInterfaces framework:

  • CATIPLMNavReference from a PLM Product Reference.
  • CATIPLMNavInstance from a PLM Product Instance.
  • CATIPLMNavRepInstance from a PLM Representation Instance.
  • CATIPLMNavRepReference ifrom a PLM Representation Reference.

Beside these generic API [3], named like that because they are implemented by several PLM Modelers, you can also use the Product APIs. Sometimes it is mandatory (like for Port), but when you have the choice, prefer to use generic APIs instead those of the modeler. Your code will be more recognizable whatever the current application.

Lets us expose the API to navigate between:

  • Navigation between LM Product Instance and PLM Product Reference
  • Navigation From a PLM Product Reference, PLM Product Representation Instance, and PLM Product Representation Reference
  • Navigation between PLM Product Reference and PLM Port

Navigation between PLM Product Instance and PLM Product Reference

  1. (1P) From a child to its father: CATIPLMNavInstance::GetFather or CATIPrdObject::GetFatherProduct

    These methods applied on a PLM instance retrieve its aggregating reference

    Apply on [Fig.3] Return values
    CAAPstAxleAss.1 CAAPstSkateBoard
  2. (2P) From a father to its child: CATIPLMNavReference::ListChildren (*) - CATIPLMProducts::Iterator

    These methods applied on a PLM Reference retrieve its direct childrens.

    Apply on [Fig.3] Return values
    CAAPstSkateBoard CAAPstAxleAss.1 & CAAPstAxleAss.2

    (*) Listchildren filtered with PLMCoreInstance.

  3. (3P) From an instance to its reference: CATIPLMNavInstance::GetReferenceInstanceOf - CATIPrdObject::GetReferenceObject

    These methods applied on a PLM instance retrieve the Reference from which the instance comes.

    Apply on [Fig.3] Return values
    CAAPstAxleAss.1 CAAPstAxleAss
  4. (4P) From a reference to its instances(s) - CATIPLMNavReference::ListInstances - CATIPrdObject::GetAllInstances.

    These methods applied on a PLM reference retrieve all its "loaded" PLM instances.

    Apply on [Fig.3] Return values
    CAAPstAxleAss CAAPstAxleAss.1 & CAAPstAxleAss.2

    See [Fig.5] to understand what does "loaded" mean.

Navigation From a PLM Product Reference, PLM Product Representation Instance, and PLM Product Representation Reference

  1. (1R) From a child to its father: CATIPLMNavRepInstance::GetFather - CATIPrdObject::GetFatherProduct

    These methods applied on a PLM Representation Instance retrieve its aggregating reference:

    Apply on [Fig.3] Return values
    CAAPstShortBoard.1 CAAPstSkateBoard
  2. (2R) From a father to its child: CATIPLMNavReference::ListChildren (*) - CATIPLMRepInstances::Iterator

    These methods applied on a PLM Reference retrieve its direct representation instances (PLM Representation Instance)

    Apply on [Fig.3] Return values
    CAAPstSkateBoard CAAPstShortBoard.1

    (*) Listchildren filtered with PLMCoreRepInstance.

  3. (3R) From an instance to its reference: CATIPLMNavRepInstance::GetRepReferenceInstanceOf - CATIPrdObject::GetReferenceObject

    These methods applied on a PLM Representation instance retrieve the Reference from which the instance comes.

    Apply on [Fig.3] Return values
    CAAPstShortBoard.1 CAAPstShortBoard
  4. (4R) From a reference to its instances(s): CATIPLMNavRepReference::ListInstances - CATIPsiRepresentationReference::GetRepInstances

    These methods applied on a PLM Representation reference retrieve all its "loaded" representation instances.

    Apply on [Fig.3] Return values
    CAAPstShortBoard CAAPstShortBoard.1

    This picture can help to illustrate what "loaded' means.

    Fig.5 A Product Assembly

    Consider these three cases:

    • The complete assembly is loaded, the result of GetRepInstances applied on Rep is RepInst1, RepInst2 and RepInst3.
    • Product1 is opened in ShortNavigation [4] mode. Product1, Product 2, Product 3 are in session. But since RepInst1, RepInst2 and RepInst3 are not loaded in memory, the result of GetRepInstances applied on Rep is an empty list.
    • Product1 is opened in Authoring mode (all the structure from Product1 is loaded), and Product4 in ShortNavigation mode, the result will be RepInst1, RepInst2.

Navigation between PLM Product Reference and PLM Port

There is no generic APIs to handle the publication (a PLM Port) [5]. It is inevitably a Product API.

  • The CATIPLMPublications::Iterator retrieves all the publications (PLM Port object) aggregated by a PLM Product Reference.
  • The CATIPLMPublication::GetOwner retrieves the PLM Product Reference aggregating the current publication.

Now, let's consider the interfaces and their methods to navigate on the "unfolded" view.

Navigation APIs

This section explains four kinds of navigation:

  • Inside the Occurrence Model
  • From the Occurrence Model to the Instance/Reference Model
  • From the Instance/Reference Model to the Occurrence Model
  • From an Occurrence to its Collection of PLM Representation Instances

All these kinds of navigation are done using generic APIs, those of the CATPLMComponentInterfaces framework.

  • CATIPLMNavOccurrence from an occurrence
  • CATIPLMNavReference from a PLM Product Reference.
  • CATIPLMNavInstance from a PLM Product Instance.

Except for the last case, where a Product interface is used.

Inside the Occurrence Model

  • From a parent to its child(s): CATIPLMNavOccurrence::ListChildren

    This method returns all the direct occurrences of the given occurrence.

    Apply on [Fig.6] Return values
    "Rear Axle-Left Wheel" "Rear Axle-Left Wheel - Screw 1", "Rear Axle-Left Wheel - Screw 2", "Rear Axle-Left Wheel - Screw 3"
    "CAAPstSkateBoard" "Rear Axle" , "Front Axle"
    "Rear Axle" "Rear Axle-Left Wheel ", "Rear Axle-Right Wheel"
  • From a child to its parent: CATIPLMNavOccurrence::GetFather

    This method returns the occurrence aggregating the current occurrence.

    Apply on [Fig.6] Return values
    "Rear Axle-Left Wheel - Screw 3" "Rear Axle-Left Wheel"
    "CAAPstSkateBoard" E_FAIL
    "Rear Axle" "CAAPstSkateBoard"

From the Occurrence Model to the Instance/Reference Model

  • From an occurrence to its instance: CATIPLMNavOccurrence::GetRelatedInstance

    This method returns the instance (of the instance/reference model) associated with the current occurrence.

    Apply on [Fig.6] Return values (of [Fig.3])
    "CAAPstSkateBoard" NULL (Hresult=S_FALSE)
    "Rear Axle" CAAPstAxle.1
    "Front Axle" CAAPstAxle.2
    "Rear Axle-Left Wheel - Screw 1" CAAPstScrew.1
    "Rear Axle-Right Wheel- Screw 1" CAAPstScrew.1

    There is no sense for the root occurrence which always represents a reference in the instance/reference model.

  • From an occurence to its reference: CATIPLMNavOccurrence::GetRelatedReference

    This method returns the reference (of the instance/reference model) associated with the current occurrence.

    Apply on [Fig.6] Return values (of [Fig.3])
    "CAAPstSkateBoard" "CAAPstSkateBoard"
    "Rear Axle" CAAPstAxleAss
    "Front Axle" CAAPstAxleAss
    "Rear Axle-Left Wheel - Screw 1" CAAPstScrew

    For any screw selected among the 12 screw, the result will be the same reference. The origin's reference of the instance.

From the Instance/Reference Model to the Occurrence Model

  • From a reference, CATIPLMNavReference:: ListRelatedOccurrences

    This method returns the occurrences (of the occurrences model) associated with the current reference. The result of the method (second argument) is depending on the first argument (the context)

    • The context is a reference

    The occurrences are looked for inside the occurrence model associated with the context. Inside this occurrence model, the occurrences whose the related reference is this are returned.

    Apply on [Fig.3] Context from [Fig.3] Return values (of [Fig.6])
    CAAPstAxleAss CAAPstSkateBoard "Front Axle","Rear Axle"
    CAAPstAxleAss CAAPstAxleAss empty string

    If there is no occurrence model associated with the context (a reference), the result will be the empty string.

    • The context is an occurrence

    The occurrences are looked for among the context's children. The occurrences whose the related reference is this are returned.

    Apply on [Fig.3] Context from [Fig.6] Return values (of [Fig.6])
    CAAPstWheel "Front Axle" "Front Axle-Left Wheel",Front Axle-Right Wheel"
    CAAPstAxleAss "Front Axle" "Front Axle"
  • From an instance , CATIPLMNavInstance:: ListRelatedOccurrences

    This method returns the occurrences (of the occurrences model) associated with the current instance. The result of the method (second argument) is depending on the first argument (the context)

    • The context is a reference.

    The occurrences are looked for inside the occurrence model associated with the context. Inside this occurrence model, the occurrences whose the related instance is this are returned.

    Apply on [Fig.3] Context from [Fig.3] Return values (of [Fig.6])
    CAAPstAxle.1 CAAPstSkateBoard "Front Axle"
    CAAPstWheel.1 CAAPstSkateBoard "Front Axle-Right Wheel","Rear Axle-Right Wheel"
    CAAPstAxle.1 CAAPstAxleAss empty list

    In the context CAAPstAxleAss, the result is an empty list because the occurrence model does not exist for this reference in your example, but it can exist.

    • The context is an occurrence

    The occurrences are looked for among the context's children. The occurrences whose the related instance is this are returned.

    Apply on [Fig.3] Context from [Fig.6] Return values (of [Fig.6])
    "Front Axle" "Front Axle" "Front Axle"
    CAAPstWheel.1 "Front Axle" "Front Axle-Right Wheel"

From an Occurrence to its Collection of PLM Representation Instances

Imagine that you want to retrieve from "Front Axle-Left Wheel" [Fig.6] the collection of PLM Rep Instances just below as depicted by this interactive view.

There are two means:

  1. From the occurrence retrieve the Related Reference (the Wheel reference), and then use CATIPLMRepInstances.
  2. From the occurrence, use CATIPLMRepInstances.

In Short

This article has exposed how to navigate:

  • On the "Instance/Reference" Model. See the use case [7].
  • On the "Occurrence" Model. See the use case [8].
  • Between the two models. See the use case [8].

It is strongly recommended to use the Generic interfaces:

  • CATIPLMNavReference, CATIPLMNavInstance, CATIPLMNavRepReference, CATIPLMNavRepInstance for instance/reference model navigation
  • CATIPLMNavOccurrence for occurrence model navigation.

In the instance/reference model navigation, there are some Product interfaces:

  • You must use them for Port since there is no generic API for Port.
  • There is no apis for connection, you should refer to the application's API.
  • CATIPLMProducts, CATIPrdObject, CATIPLMRepInstances, CATIPrdAggregatedRepresentations may be used.
