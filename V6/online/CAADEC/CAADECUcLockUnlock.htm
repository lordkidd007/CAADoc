The LockUnlock Command

Typically, the lockunlockEx command is realized by invoking the 3DEXPERIENCE Platform Save command in the interactive session, as depicted in the fig. that follows. In other words, 3DEXPERIENCE Platform Save is the UI equivalent of checking in a document to 3DEXPERIENCE Platform at the backend.

To modify a design, you should lock the CAD object. Locking an object does not make it read-only; it only specifies whether or not you can check a file into the object. Unlocking an object allows other users to modify the design.

You often lock objects when checking them out (see Checking Out CAD Objects).

lockunlockEx Request XML

The lockunlockEx request XML must contain the object name, type (cadtype), filename and dirname for each cad object to be Loked/Unlocked. The table below lists the attributes in this request and their behavior on request.

Attribute Name Description
cname should be lockunlockEx (Mandatory)
allowownerunlock This is a non mandatory Boolean attribute. If true, then the owner can unlock the object even if that is locked by some other user provided the owner has unlock access. In other cases its value is treated as false.
busid Business object id of the file. Not mandatory, if not present type and name will be used to recognize the file.
name Name of the business object to be locked/unlocked.
filename Name of the file that exists in the local disk.
action Action should specify command. It should either lock or unlock (Mandatory).
dirname Directory name where the file exists (Mandatory)
type This attribute specifies the CAD type of the file. (Mandatory)
islocked If file is already locked by user then value will be set as true. If file is not locked by any user then value will set as false.
refreshlist Contains a ‘|’ separated list of BusinessObject IDs which needs to be refreshed in the CSE Open dialog. This is in addition to the ‘busid’ attribute. The ID present in the ‘busid’ attribute will NOT be present in this list. This is present in the response only of ‘output’ node is present in request.
mxtype Contains the 3DEXPERIENCE Platform Type of the major object. Sent by X-CAD in the response incase of a failure. The CSEs can use this to display a consolidated message window incase of failure.
mxversion Contains the 3DEXPERIENCE Platform Revision of the major object. Sent by X-CAD in the response incase of a failure. The CSEs can use this to display a consolidated message window incase of failure.

Error handling in lockunlockEx:

Few error codes in lockunlockEx send additional information along with the message. The additional information is present in “errorinfo” node below the “info” nodes of the response. The error will be thrown for the first object to fail. A sample xml is provided at CAADECIntegration.edu\InputData\lockunlockExFailureResponseWithAdditonalErrorInformation.xml.
